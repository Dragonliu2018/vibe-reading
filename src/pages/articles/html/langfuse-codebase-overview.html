项目简介

什么是 Langfuse？

Langfuse 是专为 LLM 应用设计的可观测性平台，提供追踪、提示词管理、评测、数据集管理等全生命周期工具。 支持自托管或托管云服务，使用 MIT 协议开源。

🔭

LLM 可观测性

全链路追踪 LLM 调用、Span、Event，支持复杂 Agent 调试与用户会话分析。

📝

提示词管理

版本控制、协同迭代提示词，强缓存机制保证零延迟线上更新。

🧪

评测系统

LLM-as-Judge、代码评测、人工标注、用户反馈，多维度评测框架。

🗃️

数据集管理

管理测试集、运行 A/B 实验，构建持续评测的黄金数据集。

设计哲学

架构原则

来自 .agents/ARCHITECTURE_PRINCIPLES.md — 这些原则指导 Langfuse 所有技术决策， 以高规模、可探索的可观测性为目标。

1

以观测（Observation）为主要分析单元

Trace 只是关联相关 Observation 的协关句柄，而非唯一入口。系统围绕 Observation 构建分析，而不是强制从 Trace 顶层开始。

2

宽事件（Wide Events）优于碎片化记录

用富含属性的宽事件替代分散的 Metrics、Logs 和 Traces，避免后期重建上下文的昂贵开销。

3

保留高基数上下文

让用户可以对任意维度进行切片、分组、过滤，而无需预定义每个未来问题。不在摄取时丢弃元数据。

4

不可变/追加式事件记录

高频遥测数据优先使用不可变记录。需要读时去重的更新会在规模化时产生隐性查询代价。

5

谨慎反范式化（Denormalization）

仅在能消除热路径 Join 并将常见过滤条件变为直接列谓词时才进行反范式化，避免过度设计。

6

列式存储优先设计

窄字段选择、时间区间扫描、有效排序键、数据裁剪。列表/仪表盘/聚合视图基于紧凑表示，大原始 Payload 仅在详情页按需加载。

7

API 契约需具备规模意识

必要时要求时间窗口参数，暴露字段选择，使用 Token 分页，避免扫描全部历史数据的默认行为。

8

成本与运维简单性是架构约束

额外的数据库、队列、物化视图、迁移脚本都必须证明其长期运维价值，而不是轻易引入。

9

保留实时/近实时调试能力

批处理可以辅助，但不能让新鲜的生产行为对用户不可见。可观测性必须是即时的。

const ObservationSchema = z.object({

  startTime:            z.date(),
  endTime:              z.date().nullable(),
  completionStartTime:  z.date().nullable(),

  // LLM 调用信息 (GENERATION 类型)
  model:                z.string().nullable(),
  modelParameters:      JsonNullable,
  input:                JsonNullable,
  output:               JsonNullable,

  // Token 用量与成本
  inputUsage:           z.number().nullable(),
  outputUsage:          z.number().nullable(),
  totalUsage:           z.number().nullable(),
  inputCost:            z.number().nullable(),
  outputCost:           z.number().nullable(),
  totalCost:            z.number().nullable(),

  // 告警与状态
  level:                ObservationLevel,       // DEBUG | DEFAULT | WARNING | ERROR
  statusMessage:        z.string().nullable(),

  // 关联信息
  parentObservationId:  z.string().nullable(),  // 支持嵌套 Span
  promptId:             z.string().nullable(),  // 关联提示词
  promptName:           z.string().nullable(),
  promptVersion:        z.number().nullable(),

  // 工具调用 (TOOL / AGENT 类型)
  toolDefinitions:      JsonNullable,
  toolCalls:            JsonNullable,

  // 元数据
  metadata:             JsonNullable,
  tags:                 z.array(z.string()),
  version:              z.string().nullable(),
});
数据层

双数据库存储架构

Langfuse 采用 PostgreSQL + ClickHouse 双引擎架构，按数据特性分别存储，兼顾一致性与大规模分析性能。

🐘 PostgreSQL — 关系型数据
✓ 用户、组织、项目（多租户元数据）
✓ 提示词版本、评测配置、数据集定义
✓ API Keys、LLM API Keys
✓ 标注队列、仪表盘、Webhook 配置
✓ Prisma ORM 管理，支持在线迁移
🏛️ ClickHouse — 列式分析数据
✓ Traces — 高频追踪记录
✓ Observations — 海量观测事件
✓ Scores — 评分时序数据
✓ 时间范围扫描、列投影、ORDER BY 优化
✓ 追加写入，不可变事件模型
⚡ Redis 承担三个角色：BullMQ 任务队列 · 应用层缓存 · NextAuth.js 会话存储

部署方式

Langfuse 支持多种部署模式，从本地 Docker 到生产级 Kubernetes 集群。

🐳

Docker Compose

最简单的自托管方式。一条命令启动完整栈（Langfuse + PostgreSQL + ClickHouse + Redis），适合开发和小规模部署。

docker compose up -d
☸️

Kubernetes / Helm

官方提供 Helm Chart，支持水平扩展。Web 和 Worker 可独立扩缩容，适合生产环境大规模部署。

HPA 自动扩缩 多副本 Worker ConfigMap 配置
☁️

Langfuse Cloud

官方托管服务，零运维。自动扩展、SOC2 合规、企业级 SLA，支持多区域数据驻留（US/EU）。

零运维 SOC2 多区域
环境变量配置（核心）
# 数据库
DATABASE_URL="postgresql://..."
CLICKHOUSE_URL="http://..."
REDIS_HOST="localhost"

# 认证
NEXTAUTH_URL="https://your-domain.com"
NEXTAUTH_SECRET="..."

# 对象存储（可选）
LANGFUSE_S3_BUCKET="..."
AWS_REGION="us-east-1"

# 企业版（可选）
LANGFUSE_EE_LICENSE_KEY="..."
