项目概述
#overview · 产品定位与目录结构

Multica 是一个 AI 原生任务管理平台，专为小型团队设计。它的核心创新在于将 AI 编程智能体（Agent）提升为一等公民团队成员——Agent 可以拥有 Issue、发表评论、变更状态，与人类成员完全对等。平台支持 Web、Desktop（Electron）、Mobile（iOS）三端，后端采用 Go + PostgreSQL，前端以 Next.js 为核心并通过 packages/core 和 packages/views 实现跨端共享。

核心定位
Multica 不只是一个带 AI 功能的项目管理工具，而是将 AI 智能体重新定义为团队成员的工作方式。通过 Autopilot 调度系统，智能体可以定期自主执行任务；通过 Skills 系统，智能体可以积累特定领域能力；通过 Daemon 架构，智能体在用户本地机器上安全执行，保护代码隐私。
🤖
Agent 即成员
AI 智能体与人类成员并列，可接受 Issue 指派、发表评论、变更任务状态。assignee_type 字段区分人类和 Agent，UI 层统一处理。
📋
Issue 驱动
所有工作以 Issue 为中心流转。支持 6 种状态、4 级优先级、父子层级、多态指派、富文本评论和 Inbox 通知。
⚡
Autopilot 调度
定时（Cron）、Webhook、手动触发三种方式自动触发 Agent 执行任务。支持 issue_created 和 run_only 两种执行模式。
🛠
Skills 系统
内置技能（Git、代码审查、测试）+ 用户自定义技能 + 本地工具自动检测。技能可在多个 Agent 和 Autopilot 之间复用。
🏢
Squad
多个 Agent 组成 Squad，由 Leader Agent 统一接收任务并内部路由分配。Squad 本身可作为 Issue Assignee。
🌍
多平台覆盖
Web（Next.js）、Desktop（Electron）、Mobile（Expo iOS）三端，通过 packages/core 和 packages/views 共享 90% 以上业务逻辑。

整体架构
#architecture · 分层架构与数据流

Multica 采用清晰的分层架构。客户端（Web/Desktop/Mobile）通过 HTTP REST 进行 CRUD 操作，通过 WebSocket 接收实时事件。本地 Daemon 也通过 WebSocket 长连接与后端通信，接收任务、上报日志。后端通过 SQL 访问 PostgreSQL，通过缓存层访问 Redis，通过 SDK 访问 S3 对象存储。

架构关键决策
Daemon 与后端之间采用 WebSocket 长连接而非轮询，实现亚秒级任务分发和实时日志流式上报。所有 Agent 执行发生在用户本地机器，后端仅负责任务调度和状态同步，代码和上下文不经过云端服务器。

Handler 层
负责 HTTP 请求解析、参数验证、权限检查，调用 Service 层后返回 JSON 响应。

UUID 安全规则：
· 路径参数可能是 UUID 或可读 ID，必须经过 loadIssueForUser / loadAgentForUser 等 loader 解析
· 纯 UUID 输入使用 parseUUIDOrBadRequest()，解析失败立即返回 400
· sqlc 返回的 UUID 使用 parseUUID()（panic on invalid）
· Handler 外部使用 util.ParseUUID() 并检查 error
⚙️
Service 层
封装核心业务逻辑，调用 sqlc 生成的查询函数，触发实时事件推送。

关键服务模块：
· IssueService — 状态流转、指派逻辑、Inbox 生成
· AgentService — Agent 生命周期、状态同步
· AutopilotService — 触发调度、执行记录
· DaemonService — Daemon 注册、任务分发、日志收集
· SkillService — 内置技能加载、自定义技能 CRUD
sqlc 工作流
所有数据库查询必须先写 SQL 文件（pkg/db/queries/），然后执行 make sqlc 重新生成类型安全的 Go 代码（pkg/db/generated/）。禁止手动编辑生成目录中的任何文件。修改 SQL schema 后同样需要先创建迁移文件，再运行 sqlc。

多态 Assignee 设计
issues 表通过 assignee_type + assignee_id 组合实现多态指派。当 assignee_type = 'member' 时，assignee_id 指向 workspace_members 表；当 assignee_type = 'agent' 时，指向 agents 表。这种设计允许人类和 AI 成员在同一模型中统一处理，无需 JOIN 两张指派表。

API 设计
#api · RESTful + WebSocket 双协议

Multica 后端采用 RESTful + WebSocket 双协议设计。REST API 负责 CRUD 操作，WebSocket 负责实时事件推送与 Daemon 通信。所有工作区相关端点均通过 URL 路径中的 {wsId} 进行作用域隔离，并配合 X-Workspace-ID 请求头进行双重校验。

领域 端点 说明
Issues GET /api/v1/w/{wsId}/issues 获取 Issue 列表（支持筛选、分页）
Issues POST /api/v1/w/{wsId}/issues 创建新 Issue
Issues PATCH /api/v1/w/{wsId}/issues/{id} 更新 Issue（标题/状态/优先级/指派人）
Issues DELETE /api/v1/w/{wsId}/issues/{id} 删除 Issue
Issues GET /api/v1/w/{wsId}/issues/{id}/comments 获取 Issue 评论列表
Agents GET /api/v1/w/{wsId}/agents 获取工作区 Agent 列表
Agents POST /api/v1/w/{wsId}/agents 创建新 Agent
Agents PATCH /api/v1/w/{wsId}/agents/{id} 更新 Agent 配置
Autopilots GET /api/v1/w/{wsId}/autopilots 获取 Autopilot 列表
Autopilots POST /api/v1/w/{wsId}/autopilots/{id}/trigger 手动触发 Autopilot
Daemon POST /api/v1/daemon/register Daemon 注册（返回连接令牌）
Daemon WS /api/v1/daemon/ws WebSocket 长连接（任务分发 + 实时日志）
Daemon GET /api/v1/daemon/tasks/next 轮询下一个待执行任务
Auth POST /api/v1/auth/login 邮箱 + 密码登录（返回 JWT）
Auth POST /api/v1/auth/logout 登出（清除 cookie）
API 兼容性防御
已安装的桌面客户端可能在后端升级后继续运行旧版本，因此所有端点响应必须通过 Zod schema + parseWithFallback() 进行防御性解析，不能将网络 JSON 直接 cast 为 TypeScript 类型。新增字段时必须保持向后兼容，枚举值新增需要 default 分支处理。
🖥
本地 Daemon
#daemon · Agent 本地执行环境

本地 Daemon 是 Multica 的核心安全设计之一。所有 Agent 的 AI 代码执行发生在用户本地机器，Daemon 进程通过 WebSocket 与云端后端保持长连接，接收任务并在本地调用 Agent CLI（Claude Code、Codex 等）执行，将执行日志实时上报后端。代码、密钥、本地文件系统从不离开用户机器。

🔗
WebSocket 长连接
Daemon 启动后立即向后端注册并建立 WebSocket 连接。支持心跳保活、自动断线重连、消息队列。连接断开期间任务自动挂起，重连后恢复。
📋
任务分发
后端通过 WebSocket 推送任务到 Daemon，Daemon 根据 runtime_config 选择对应的 Agent CLI，在本地子进程中执行。支持任务队列和并发上限控制。
📡
实时日志流
Agent CLI 的标准输出逐行捕获并实时上报后端，前端通过 WebSocket 接收并渲染执行日志流。用户可实时观察 Agent 的思考和执行过程。
🔍
环境检测
Daemon 启动时自动扫描本地环境，检测已安装工具（npm/yarn/pnpm、git、docker、python、go、rust 等），上报可用工具列表作为 Agent Skills 的前提条件。
🔒
隔离执行
每个任务在独立的工作目录中执行，支持配置白名单命令和文件系统访问范围。Agent 的执行上下文不共享，避免跨任务的意外状态污染。
⚡
多运行时支持
通过统一的 runtime_config 抽象，支持 10+ 种 Agent CLI。用户可在 Agent 配置中切换运行时，Daemon 自动适配不同 CLI 的调用协议。

桌面应用
#desktop · apps/desktop · Electron
🗺
路由分类
Session Routes： 工作区内标签页目标，如 /:slug/issues，通过 react-router-dom 管理。

Transition Flows： 单次预工作区操作（创建工作区、接受邀请），通过 WindowOverlay 实现，非路由。禁止向 routes.tsx 添加此类流程。

Error/Stale States： 过时工作区标签页自动愈合，丢弃过时标签组，不显示桌面错误页。
🔧
桌面特有功能
DragStrip 拖拽区域： 全窗口视图必须挂载 <DragStrip /> 作为首个 flex 子节点，顶部 48px 内的交互控件需设置 WebkitAppRegion: "no-drag"。

setCurrentWorkspace： 工作区切换的唯一可信来源，离开工作区时必须显式调用 setCurrentWorkspace(null, null)。

WindowOverlay： 预工作区流程通过此机制实现，避免路由污染。

跨工作区导航： 必须通过 NavigationAdapter.switchWorkspace() 实现。
共享逻辑
桌面应用通过 packages/core · packages/ui · packages/views 与 Web 共享所有业务逻辑，仅在路由（react-router-dom vs Next.js router）、Electron API（原生菜单、文件系统）、窗口管理（DragStrip、WindowOverlay）处存在差异。
📱
移动端
#mobile · apps/mobile · Expo / React Native iOS

移动端（Expo · React Native iOS）相对独立，仅共享 @multica/core 中的类型定义和纯函数。移动端有独立的 UI 组件层、状态管理、i18n、构建流水线和发布节奏。在触碰 apps/mobile/ 之前，必须先阅读 apps/mobile/CLAUDE.md 中的强制预检流程。

🎨
UI 组件水位
移动端不使用 packages/ui 的 Web 组件，而是维护独立的 NativeWind 4 + Tailwind 3.4 组件库。组件命名与语义和 Web/Desktop 保持一致，实现有差异时以手机交互最优为准。
📡
实时策略
独立的 WebSocket 客户端实现，轮询策略和重连逻辑针对移动网络优化。不依赖 packages/core 的 ws-client.ts，避免浏览器/Node.js 特定 API。
💾
存储
使用 Expo SecureStore 管理认证令牌，AsyncStorage 持久化用户偏好和草稿。不使用 localStorage（浏览器 API），通过 StorageAdapter 抽象层访问。
端对端一致性原则
移动端必须与 Web/Desktop 保持产品语义一致：计数方式、权限检查、枚举/状态流转、数据标识。UI 和交互可因手机场景而有差异，但业务逻辑不能背离。Issue 状态枚举、Agent 状态机、优先级定义必须与后端完全匹配。
🧩
packages/core
#core · 无头业务逻辑核心

packages/core 是整个系统的无头业务逻辑核心。它不依赖任何 UI 框架（无 react-dom）、不访问浏览器 API（无 localStorage，通过 StorageAdapter 抽象）、不读取环境变量（无 process.env），可在 Web、Desktop、Mobile（受限）三端复用。

🌐
api/ 目录
client.ts — 统一 API 客户端，基于 fetch 封装，自动注入 workspace header，处理认证 token 刷新。

ws-client.ts — WebSocket 客户端，支持心跳保活、断线重连、消息队列，实现 exponential backoff。

schema.ts — Zod 响应 Schema 集合 + parseWithFallback() 防御解析函数，所有网络 JSON 必须经过此层。
📦
issues/ · agents/ · autopilots/
每个领域包含：
· Query key factories（workspace-scoped，含 wsId）
· Mutation hooks（乐观更新 → 失败回滚 → settle 后 invalidate）
· Zustand stores（过滤器、草稿、UI 状态）
· WebSocket updaters（收到实时事件后 patch Query cache）
📝
types/ 目录
约 25 个类型文件，覆盖系统所有领域：Agent、Issue、Workspace、Member、Autopilot、Event、Skill、Squad、Inbox、Permission、FeatureFlag 等。类型定义是前后端协议的唯一真相来源，通过 Zod schema 在运行时验证。
🚩
feature-flags/ · permissions/
框架级别的 Feature Flag 评估系统（MUL-3615），支持按工作区、按用户、按环境启用功能。权限检查逻辑（基于 workspace role + resource ownership）。两者均是纯函数，无副作用，可在服务端和客户端复用。
🎨
packages/ui
#ui · 原子 UI 组件库

packages/ui 是纯原子 UI 组件库。硬性约束：零业务逻辑、零 @multica/core 导入、只输出视觉原语。基于 shadcn + Base UI 构建，通过 pnpm ui:add <component> 添加新组件。

组件列表
Button Badge Avatar Card Dialog Dropdown Input Textarea Select Checkbox Switch Tooltip Popover Toast Table Tabs Accordion Calendar Command Form Label Progress Separator Sheet Skeleton Slider Spinner ScrollArea Resizable
CSS 设计令牌
UI 样式基于语义令牌（bg-background · text-muted-foreground · border-border），避免硬编码 Tailwind 颜色类（如 bg-gray-800）。样式文件在 packages/ui/styles/ 中集中管理，Web 和 Desktop 共享同一套样式基础。新增组件必须使用语义令牌，不得引入硬编码颜色。
📄
packages/views
#views · Web + Desktop 共享业务页面

packages/views 是 Web 和 Desktop 共享的业务页面和组件层。它可以使用 React、packages/core、packages/ui，但不能使用 Next.js（next/*）、react-router-dom，也不包含 Zustand stores（stores 在 packages/core 中）。所有导航使用 NavigationAdapter 抽象。

✏️
editor/
基于 TipTap 的富文本编辑器，支持 Markdown、@提及、附件上传。Web 和 Desktop 共享同一编辑器实例。支持 Agent 和人类成员的 @提及补全。
📋
issues/
Issue 列表、看板、详情、评论区。包含状态筛选器、优先级图标、Assignee 选择器（区分 member/agent）。支持拖拽排序和批量操作。
🤖
agents/
Agent 详情页、执行日志流、技能列表、状态指示器。实时日志通过 WebSocket 流式渲染，支持高亮显示 Agent 的思考链输出。

NavigationAdapter 模式
packages/views 中所有导航使用 useNavigation().push() 或 <AppLink>，通过适配器抽象屏蔽 Next.js router 和 react-router-dom 的差异，实现一份视图代码同时运行在 Web 和 Desktop 上。适配器由各 app 层注入（NavigationProvider），views 层永远不感知底层路由实现。

Agent 系统
#agents · 状态机 · 可见性 · Squad
🔄
Agent 状态机
idle 空闲，等待任务分配
working 正在执行任务，Daemon 活跃
blocked 等待用户输入或外部条件
error 执行失败，需要人工介入
offline Daemon 未连接
👁
可见性控制
workspace： 工作区全员可见可用，适合团队共享的通用 Agent（如代码审查 Agent、测试生成 Agent）。

private： 仅创建者可见，适合个人工作流的专属 Agent，其他成员不可查看或使用。

runtime_mode：
· local 本地 Daemon 执行，代码不离开用户机器
· cloud 云端执行（规划中）
🏢
Squad（Agent 组）
Squad 是多个 Agent 的聚合单元，由一个 Leader Agent 负责接收任务并路由给成员。适合前端组、后端组等专项 Agent 团队。Squad 本身可作为 Issue Assignee，由 Leader 进行内部分配。max_concurrent 字段控制并发执行数量上限，防止资源耗尽。Leader 可根据任务内容（关键词匹配、技能标签）自动决定分配给哪个成员 Agent。

Autopilot 调度系统
#autopilots · 触发器 · 执行模式

Autopilot 是 Multica 的自动化调度引擎，允许用户设置规则让 Agent 在特定条件下自动执行任务。支持多种触发方式和执行模式的组合，实现真正的 AI 自动化工作流。

⏰
触发器类型
Cron 触发
标准 cron 表达式定时触发（如 "0 9 * * 1-5" 每工作日 9 点）
Webhook 触发
外部系统通过 HTTP 请求触发 Autopilot，支持 GitHub、Slack 等集成
手动触发
通过 UI 或 API 立即触发一次执行
Issue 创建触发
当符合条件的 Issue 被创建时自动触发
🎯
执行模式
issue_created
触发后自动创建一个新 Issue，并将其分配给指定 Agent 执行。适合定期报告、定期检查类任务。
run_only
直接在已有上下文中运行 Agent，不创建新 Issue。适合维护、监控、自动修复类任务。
Assignee
可以是单个 Agent 或一个 Squad（由 Leader 内部路由分配给最合适的成员 Agent）
🛠
Skills 系统
#skills · 内置 · 自定义 · 本地检测

Skills 系统为 Agent 提供可复用的能力模块。技能定义了 Agent 应该如何处理特定类型的任务，包括触发条件、执行指令和参考材料。技能可以跨 Agent 和 Autopilot 复用，随时间不断积累和迭代。

📦
内置技能
随平台预打包的技能，存储在 server/internal/service/builtin_skills/ 中，每个技能由 YAML 配置 + Markdown 参考资料组成。涵盖 Git 操作、代码审查、测试编写、文档生成等常见开发工作流。修改内置技能时需同步更新对应的 SKILL.md 和 references/*-source-map.md。
✏️
用户自定义
用户可在工作区内创建自定义技能，定义技能名称、触发条件、执行指令和上下文材料。技能可在多个 Autopilot 和 Agent 之间复用，随时间积累和迭代。技能版本化管理，修改前保留历史快照。
🔍
本地检测
Daemon 启动时自动扫描本地环境，检测已安装工具（npm/yarn/pnpm · git · docker · python · go · rust 等），将可用工具列表上报后端作为 Agent 的可用 Skills 前提条件。后端据此筛选该 Agent 可以执行的技能类型。

实时同步
#realtime · WebSocket 事件驱动架构

Multica 通过 WebSocket 实现全实时 UI 更新。后端推送事件到前端，前端通过 TanStack Query cache patching 或 invalidation 更新 UI，无需轮询。所有在线成员即时看到同一状态，实现真正的实时协作体验。

事件类型（80+ 种）
issue.created issue.updated issue.deleted issue.assigned comment.created agent.status_changed agent.task_started agent.task_completed agent.task_failed autopilot.triggered autopilot.completed daemon.connected daemon.disconnected workspace.member_joined inbox.created
Cache 更新策略
收到 WebSocket 事件后，前端通过两种策略更新状态：
(1) 局部 patch — 直接修改 Query cache 中的对应数据，适合已知字段的增量更新（如 issue.updated 只更新 status 字段）。
(2) 完整 invalidate — 标记查询为过时并触发后台 refetch，适合复杂变更（如成员加入后权限变化影响多个查询）。

WebSocket 事件永远不直接写入 Zustand store，只通过 Query cache 间接影响 UI，确保服务端状态和客户端状态的清晰边界。
🗂
状态管理架构
#state · TanStack Query + Zustand 双层状态
🌐
TanStack Query（服务端状态）
管理所有需要与服务器同步的数据：
Issues Comments Members Agents Autopilots Skills Workspace Inbox
Query Key 规则： 所有 workspace-scoped 查询必须包含 wsId，确保切换工作区时缓存正确隔离。

Mutation 原则： 乐观更新 → 请求失败回滚 → settle 后 invalidate。

WebSocket 集成： 事件触发 patch 或 invalidate，不直接写 Zustand。
📦
Zustand（客户端状态）
管理纯客户端的 UI 和交互状态：
· workspace-store — 选中的工作区
· filter-store — 过滤器和搜索条件
· draft-store — 草稿内容（持久化）
· modal-store — Modal 开关状态
· tab-layout-store — 标签页布局（Desktop）
· navigation-history-store — 导航历史
Selector 规则： 必须返回稳定引用，使用 shallow 比较避免不必要渲染。禁止从 selector 返回新分配的对象/数组。

持久化： 偏好设置和草稿持久化，服务端数据和临时 UI 状态不持久化。

测试策略
#testing · 测试分层与边界规则
测试层次 工具 位置 覆盖范围
单元测试 Vitest + @testing-library/react packages/core/*.test.ts Hooks · Stores · 纯函数 · Schema 验证
组件测试 Vitest + @testing-library/react packages/views/*.test.tsx 业务组件 · 表单 · Modal · 页面
平台测试 Vitest apps/web/*.test.tsx Next.js 特有：cookies · redirects · search params
端到端测试 Playwright e2e/*.spec.ts 完整用户流程，使用 TestApiClient 做数据准备
后端测试 Go 标准测试 server/ Handler · Service · DB 查询
测试边界规则
(1) 共享组件行为禁止在 app 层测试文件中测试；
(2) packages/views/ 测试不能 mock next/* 或 react-router-dom；
(3) Mock @multica/core stores 时使用 Zustand callable-store 形式（selectorFn + getState）；
(4) Mock @multica/core/api 用于 API 调用；
(5) 新增端点响应变更时，必须添加 malformed-response 测试验证 parseWithFallback 的防御行为。

部署方式
#deployment · 三种部署模式 + 开发命令
☁️
Multica Cloud
官方托管 SaaS 版本，零运维。用户只需注册账号即可使用，无需部署任何服务器。Agent Daemon 运行在用户本地机器，数据存储在 Multica 托管的 PostgreSQL 上。
推荐用于个人和小团队
🏠
完整自托管
部署完整后端服务（Go Server + PostgreSQL + Redis + S3）。适合有合规要求或希望完全控制数据的团队。提供 Docker Compose 配置，支持一键启动。
适合企业和数据敏感场景
🔧
CLI Only
仅安装本地 Daemon CLI，连接到 Multica Cloud 或自托管后端。适合只想在本地运行 Agent 的用户，通过 Homebrew 安装：brew install multica/tap/multica-daemon
