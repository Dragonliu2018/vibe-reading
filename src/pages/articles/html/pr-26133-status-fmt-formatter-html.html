[PR-26133] 为 Status 类实现 fmt 格式化支持

通过 fmt::formatter<doris::Status> 模板特化，让 Apache Doris BE 的核心错误类型 原生支持 fmt::format("{}", status)，消除 BE 代码中无处不在的 .to_string() 样板调用。

🐛问题背景

在 Apache Doris 的 BE（Backend）层，Status 是贯穿整个执行链路的核心错误处理类型—— 查询执行、存储读写、内存管理等几乎所有组件都返回 Status 以传递成功或失败信息。 然而，当开发者需要将 Status 对象格式化进日志或错误消息时，却必须手写一个令人烦躁的转换调用。

旧写法（繁琐）

C++ · 问题写法 
// 每次都要手动调用 to_string()
LOG(WARNING) << fmt::format(
    "operation failed: {}",
    status.to_string()  // ← 样板代码
);

// 或者更冗长的形式
std::string msg = fmt::format(
    "scan error: {}",
    status.to_string()
);

期望写法（简洁）

C++ · 目标写法 
// 直接传 status，像 int/string 一样自然
LOG(WARNING) << fmt::format(
    "operation failed: {}",
    status  // ← 简洁直观
);

// 格式化进任意字符串
std::string msg = fmt::format(
    "scan error: {}",
    status
);

Issue #25974 由社区成员 platoneko 提出，核心诉求是：为 Status 类注册 fmt 格式化器， 使其像内置类型一样可以直接传入 fmt::format，无需每次额外调用 .to_string()。

💡
为什么 .to_string() 是问题？ BE 代码中 Status 出现频率极高，每一处日志、错误传播都要写 .to_string()， 这不仅增加了代码噪声，更容易因为遗漏而导致日志中出现无法解析的对象地址。

📐fmt 格式化机制

{fmt} 是 Apache Doris BE 层采用的高性能格式化库（C++20 std::format 的前身）。 它通过 模板特化 机制支持用户自定义类型：只需在 fmt 命名空间下特化 fmt::formatter<T> 结构体，该类型就能在所有 fmt::format / fmt::print 调用中被识别。

formatter<T> 的两个必须方法

方法签名职责
parse() auto parse(ParseContext& ctx) 解析格式说明符（{:xxx} 中的 xxx）。不需要额外选项时直接返回 ctx.begin()。
format() auto format(T const& val, FormatContext& ctx) 执行实际格式化，将结果写入 ctx.out() 迭代器并返回新位置。

编译期解析流程

fmt::format("{}", status)
→
编译器查找 fmt::formatter<doris::Status>
→
特化存在 → 调用 parse() 验证格式说明符
→
运行时调用 format() → status.to_string() → 写入输出
→
编译期完成类型检查，无运行时虚函数开销
⚠️
特化必须放在全局命名空间 fmt::formatter 的特化需要写在 doris 命名空间之外（即全局作用域或 fmt 命名空间内）， 否则 ADL（Argument-Dependent Lookup）无法找到它。

⚙️核心实现

PR #26133 的改动极为精简：在 be/src/common/status.h 文件的命名空间闭合括号之后， 追加了 14 行 fmt::formatter 特化代码。选择放在 status.h 末尾是因为： 任何 #include "common/status.h" 的翻译单元都会自动获得格式化能力，无需额外引入新头文件。

be/src/common/status.h C++ · +14 行 
// } // namespace doris  ← 命名空间在此闭合

// specify formatter for Status
template <>
struct fmt::formatter<doris::Status> {
    // parse() — 解析格式说明符
    // 返回 ctx.begin() 表示不支持 {:xxx} 形式的额外选项
    template <typename ParseContext>
    constexpr auto parse(ParseContext& ctx) {
        return ctx.begin();
    }

    // format() — 执行实际格式化
    // 委托给 Status::to_string()，写入 ctx.out() 输出迭代器
    template <typename FormatContext>
    auto format(doris::Status const& status, FormatContext& ctx) {
        return fmt::format_to(ctx.out(), "{}", status.to_string());
    }
};

关键设计决策解析

template<> 全特化
使用 template <> 对 doris::Status 进行完全特化， 而不是偏特化。精确匹配，不会与其他类型产生歧义。
零歧义
constexpr parse()
parse() 被标记为 constexpr，允许编译器在编译期完成格式串验证， 非法格式说明符直接报编译错误而非运行时崩溃。
编译期安全
委托 to_string()
format() 内部调用 status.to_string()，复用已有实现， 保证格式化结果与现有日志系统完全一致，不引入新的行为差异。
行为一致
放置于 status.h 末尾
特化代码跟随头文件一起传播，任何已经 include status.h 的文件无需额外改动即可获得格式化能力。
零侵入

fmt::format_to 的作用

fmt::format_to(ctx.out(), "{}", status.to_string()) 将格式化结果直接写入 FormatContext 提供的输出迭代器 ctx.out()，避免创建临时 std::string， 与 fmt 库的整体设计哲学一致——按需写入，不做多余分配。

🧪单元测试

PR 在 be/test/common/status_test.cpp 中新增了一个 Format 测试用例， 覆盖两种基本场景：Status::OK() 和 Status::InternalError()。 测试逻辑非常直观——验证 fmt::format("{}", status) 的结果与 fmt::format("{}", status.to_string()) 完全一致。

be/test/common/status_test.cpp C++ · +10 行 
TEST_F(StatusTest /*unused*/, Format /*unused*/) {
    // 场景 1：status == OK
    Status st_ok = Status::OK();
    EXPECT_TRUE(
        fmt::format("{}", st_ok).compare(
            fmt::format("{}", st_ok.to_string())
        ) == 0
    );

    // 场景 2：status == error（InternalError）
    Status st_error = Status::InternalError("123");
    EXPECT_TRUE(
        fmt::format("{}", st_error).compare(
            fmt::format("{}", st_error.to_string())
        ) == 0
    );
}
